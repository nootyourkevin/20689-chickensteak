# VocaLand v2 产品全流程审核

> 资深产品视角：遍历所有用户场景，标注漏洞和需要提前规范的决策。
> 严重程度：🔴 致命 / 🟡 重要 / 🟢 建议

---

## 一、逐场景遍历

### 场景 1：首次用户 — 创建角色

**流程**：打开 App → Page 1 空状态 → 点"+ 新建" → 选水平气泡 → 选兴趣气泡 → 输名字 → 创建完成

**漏洞：**

🔴 **中途退出没保护**：用户在选兴趣时关掉 App，下次打开是什么状态？
→ **规范**：角色创建要么全完成，要么全不存（事务性）。新建角色时用一个临时对象，点"完成"才写入数据库。

🟡 **"跳过"要不要给**：水平可以跳过（默认中级），兴趣也可以跳过吗？
→ **规范**：兴趣必须至少选 1 个（没兴趣就没法生成话题，产品核心循环断了）。水平可以跳过，默认"中级"。

🟡 **角色名能不能重复**：两个人都叫"小明"？
→ **规范**：允许同名。内部用 ID 区分，显示时加头像/颜色区分。

🟡 **能不能修改已创建角色的水平和兴趣？**
→ **规范**：需要。在 Page 1 角色卡片上加一个编辑按钮（齿轮图标），点击可以修改。修改后 Page 2 的话题会相应变化。

🟡 **角色数量上限？**
→ **规范**：建议上限 5 个。这是学习工具不是社交App，没必要无限建角色。

---

### 场景 2：已有角色 — 回到 App

**流程**：打开 App → Page 1 显示角色卡片列表 → 点一个角色 → 进 Page 2

**漏洞：**

🟢 **上次聊天到一半就退出了**：Page 2 要不要显示"上次你正在聊 XXX，继续？"
→ **规范**：不需要。每次进 Page 2 都是新的开始。旧的聊天已经保存了生词，聊天记录可以丢弃（或者保留最近一次聊天的摘要）。

🟡 **能不能删除角色？**
→ **规范**：需要。长按角色卡片或点编辑后出现"删除角色"。删除时弹确认框："删除 X 角色将同时删除该角色的所有学习记录，不可恢复。" 确认后物理删除该角色的所有数据。

---

### 场景 3：Page 2 — 话题选择

**流程**：看到 5-8 个基于兴趣的话题摘要卡片 → 滑看 → 选一个 → 进 Page 3

**漏洞：**

🔴 **话题生成失败怎么办**：LLM 挂了或返回垃圾数据。
→ **规范**：
- 先展示缓存的上次话题（如果有）
- 如果缓存也没有，展示一组预设的通用话题（hardcoded fallback list，每个兴趣领域 3 个）
- 显示"话题加载失败，已为你准备备用话题"的提示
- 提供"重试"按钮

🔴 **用户不想聊任何推荐话题**：需要自定义话题入口。
→ **规范**：Page 2 顶部放一个输入框 + 按钮："或者，输入你想聊的任何话题..."。用户输入后直接进 Page 3，跳过话题卡片。

🟡 **同一个话题聊过了，再点一次怎么办？**
→ **规范**：允许。每次点击都是**新会话**。生词面板重置，聊天历史从零开始。可以显示"上次聊过这个话题（2026-06-28），收集了 12 个生词"作为提示。

🟡 **"每日刷新"还是"每次打开刷新"？**
→ **规范**：每次进入 Page 2 时检查：如果距离上次生成 > 30 分钟，就重新生成。否则展示缓存。避免用户频繁进出导致重复调用 LLM（省钱）。

🟡 **话题卡片里显示什么信息？**
→ **规范**：
- 标题（不超过 40 字）
- 2-3 句摘要（不超过 80 字）
- 来源标记（"AI 生成"或新闻来源名）
- 难度标签（根据用户水平显示 Estimated difficulty: ★★☆）

🟢 **要不要标记"已读/已聊过"？**
→ **建议**：聊过的话题卡可以加一个淡淡的"已聊过"标记，但不阻止用户再聊。

---

### 场景 4：Page 2 → Page 3 转换

**流程**：点话题卡片（或输入自定义话题）→ 系统构建 Prompt → 显示聊天界面

**漏洞：**

🔴 **转换过程有延迟**：LLM 需要生成开场白。
→ **规范**：显示一个过渡状态——"Leo 正在准备这个话题..." 加上加载动画。LLM 生成开场白后再显示聊天界面。超时 15 秒则用默认开场白："Let's talk about [topic]. What do you think about it?"

🟡 **自定义话题没有摘要和背景**：
→ **规范**：自定义话题直接作为聊天主题传给 LLM。Prompt 中加一行："Your partner wants to talk about: [用户输入的话题]。Introduce this topic naturally in your first message."

---

### 场景 5：Page 3 — 聊天核心流程

**流程**：AI 发开场白 → 用户读 → 遇到生词点击 → 弹窗查释义 → 词加入右侧面板 → 用户回复 → AI 再回 → 循环

**漏洞：**

🔴 **点击取词 — Qt 技术细节**：
- QTextEdit 里每个单词都需要能用 `cursorForPosition()` 定位
- 点击后需要向前后扫描单词边界，处理标点符号（`"algorithm"` → 去掉引号 → `algorithm`）
- 缩写词（`don't`, `it's`）：点击 `don't` 应该查 `don't`，不是 `don` 或 `t`
- 数字和符号不触发查词
- **规范**：正则 `[a-zA-Z'][a-zA-Z']+` 作为单词匹配，最小长度 3 个字符

🔴 **弹窗定位**：弹窗出现在哪里？
- 鼠标附近？气泡旁边？固定位置？
- **规范**：弹窗出现在被点击气泡的旁边（屏幕空间够的话），不遮挡对话。如果是小屏幕，弹窗居中显示，背景半透明遮罩。

🔴 **查词优先级**：
1. 本地 SQLite 词典（6,664 词）→ 命中直接显示
2. 本地没命中 → 调 CloudLLM 查询（显示"查询中..."）
3. LLM 也失败 → 显示"抱歉，暂时无法获取该词的释义"

🟡 **弹窗里有什么**：
→ **规范**：
```
┌─────────────────────────────────┐
│ algorithm  /'ælɡəˌrɪðəm/  n.   │  ← 单词 + 音标 + 词性
│ ─────────────────────────────   │
│ 中文：算法；计算规则              │  ← 中文释义
│ 英文：a set of rules for        │  ← 英文释义（如果有）
│ solving a problem...            │
│ ─────────────────────────────   │
│ 📝 The algorithm can process    │  ← 例句（中英对照）
│    data faster than before.     │
│    该算法可以比以前更快地处理数据。 │
│ ─────────────────────────────   │
│ 短语：                          │  ← 短语搭配（如果有）
│ • algorithm design — 算法设计    │
│ • genetic algorithm — 遗传算法   │
│ ─────────────────────────────   │
│ [加入生词本] [关闭]               │
└─────────────────────────────────┘
```
弹窗内"加入生词本"按钮：用户点击后词才正式记录。但默认行为是——弹窗出现即自动加入生词本（减少一步操作）。"关闭"按钮关闭弹窗。提供撤销："已加入生词本 [撤销]"。

🟡 **同一个词多次点击**：
→ **规范**：第二次点击同一个词 → 直接弹窗（不重复添加到生词本），但增加该词的 `lookup_count`。右侧面板显示"出现 2 次"。

🟡 **点击的是自己发的消息里的词**（用户气泡）：
→ **规范**：一样可以点击查词。用户可能打字时用了不确定的拼写。

🔴 **生词面板更新**：
→ **规范**：每点击一个新词，右侧面板即时添加一张卡片。卡片包含：
- 单词（粗体）
- 中文释义（一行，超长截断）
- 出现次数标记
- 点击卡片可以再次打开弹窗查看详情

🟡 **"已掌握"标记**：
→ **规范**：用户可以在生词面板中长按/右键某个词 → 标记为"已掌握"（不再出现在复习队列中）。这是用户主动操作，不是系统自动判断。

### 场景 6：退出聊天 → 回到 Page 2

**流程**：用户点"← 返回" → 系统保存会话 → 回到 Page 2

**漏洞：**

🔴 **意外退出保护**：
→ **规范**：点返回按钮时，如果本轮聊天有生词：
- 弹轻提示："本次聊天已保存 12 个生词"
- 不弹确认框（太烦），直接返回
- 如果用户一个字没说就返回（0 轮对话，0 生词）→ 直接返回，不创建会话记录

🔴 **App 崩溃/被杀进程**：
→ **规范**：生词不应该只在"点返回"时才保存。应该**实时保存**——每次点击单词加入生词本时，立即写入 `user_vocabulary` 表。会话元数据（开始时间、结束时间）在退出时更新。这样即使崩溃，生词也不会丢。

🟡 **聊天记录要不要保留？**
→ **规范**：v2 第一版不保留聊天记录。每次新会话从零开始。原因：
- 简化开发（不需要聊天历史存储和加载）
- 符合产品逻辑（每次是围绕一个新话题的独立对话）
- 以后可以加"聊天记录"功能，但不是 MVP

🟡 **返回后，能不能再进入刚才的聊天？**
→ **规范**：不能。返回 = 会话结束。想聊同一个话题 → 在 Page 2 重新点 → 新会话。

### 场景 7：Page 4 — 生词复习

**流程**：Page 2 底部点"进入生词复习" → 进入 Page 4 → 卡片式复习 → 返回

**漏洞：**

🔴 **复习队列的来源**：
→ **规范**：复习队列 = 该角色所有状态为 `NEW` 或 `LEARNING` 的生词，按 SM-2 算法排序。`MASTERED` 的词偶尔抽查（10 次复习中插 1 次）。

🔴 **复习卡片的设计**：
- 正面：单词 + 音标 + 词性
- 点击翻转（或点"显示答案"按钮）
- 反面：中文释义 + 英文释义 + 例句 + 短语
- 两个按钮："没想起来" 和 "想起来了"
- SM-2 映射：
  - "没想起来" → quality=0 → 重置间隔，明天再复习
  - "想起来了" → quality=4 → 间隔翻倍

🔴 **复习会话的长度**：
→ **规范**：每次复习最多 20 张卡片。用户可以在任何时候点"结束复习"，进度保存。如果 20 张复习完还有 → 显示"今天还有 X 个词待复习，要继续吗？"

🟡 **新词什么时候进入复习队列？**
→ **规范**：生词在聊天中收集后，状态为 `NEW`。用户第一次进入复习模式时，`NEW` 词在正面展示（用户看到单词），让用户尝试回忆。如果用户完全没见过这个词（因为刚收集），可以点"没想起来"→ 明天再来。这种"先曝光、后测试"的方式符合记忆规律。

🟡 **复习完的总结**：
→ **规范**：显示本次复习统计："复习 20 词 | 记住 15 词 | 还需努力 5 词 | 累计已掌握 X 词"。

🟡 **没有词可复习时**：
→ **规范**：显示空状态插图 + "还没有收集生词，去聊个话题吧！" + "[前往话题列表]" 按钮。

---

### 场景 8：用户输入中文

**流程**：用户在聊天中打了中文（因为英语表达困难）

**漏洞：**

🔴 **系统如何处理中文输入**：
→ **规范**：
- 不拦截中文。用户想打就打。
- Prompt 中告诉 LLM："Your partner may occasionally write in Chinese when they struggle with English. When they do, gently help rephrase their message in English as part of your response, but don't make a big deal out of it."
- 中文输入中的词**不做点击取词**（因为没有英文单词可以点）
- 中文输入照样可以推进对话，不影响生词收集（生词来自 AI 的英文回复）

---

### 场景 9：水平自适应

**流程**：用户角色标记为"初级"，AI 用简单英语。但实际上用户水平提高了。

**漏洞：**

🟡 **水平不会自动调整**：
→ **规范**：v2 第一版不做自动水平调整（太复杂且容易出错）。但提供一个手动入口：在 Page 1 角色卡片编辑中可以修改水平。用户觉得对话太简单/太难时可以自己改。以后可以加自动评估，但不是 MVP。

🔴 **AI 回复的可读性**：
→ **规范**：不管什么水平，AI 回复必须遵守：
- 不要一段超过 4 句话
- 每句话不要超过 25 个词（初级）/ 40 个词（高级）
- 段落之间有空行
- 这保证了气泡不会是一大坨文字，方便阅读和点词

---

### 场景 10：网络异常

**流程**：正在聊天 → 网络断了 → CloudLLM 超时

**漏洞：**

🔴 **LLM 超时处理**：
→ **规范**：
- 发送消息后显示"Leo 正在回复..."（替代现在的 typing indicator）
- 15 秒超时 → 显示错误消息："网络似乎有问题，点击这里重试"
- 用户点"重试"→ 重新发送上一条消息给 LLM
- 重试 3 次后 → "暂时无法连接，请稍后再试。你的学习进度已保存。"

🔴 **话题生成时网络断了**：
→ **规范**：使用缓存话题或 fallback 通用话题（见场景 3）。

🟡 **单词 LLM 兜底查词时网络断了**：
→ **规范**：显示"本地词典未收录该词，且网络不可用。该词已加入生词本，联网后可查看释义。"

---

### 场景 11：数据一致性

🔴 **角色删除的级联**：
→ **规范**：删除角色时，级联删除：
- `user_profile` 中的记录
- `chat_session` 中该角色的所有会话
- `user_vocabulary` 中该角色的所有生词
- `learning_events` 中该角色的所有事件
→ 用 SQLite 外键 + `ON DELETE CASCADE`，或者代码中手动清理。

🟡 **词典更新**：
→ **规范**：现有的 `words` 表（6,664 词）是只读参考词典。不要往里面写用户数据。用户生词走 `user_vocabulary` 表。两个表互不干扰。

## 二、全局规范（需要写进代码规则的）

### UI 状态规范

| 组件 | Loading | Empty | Error | Success |
|------|---------|-------|-------|---------|
| Page 1 角色列表 | 骨架屏 | "点击+创建你的第一个学习角色" | 数据库错误提示 | - |
| Page 2 话题列表 | 骨架卡片（3 个灰色占位） | "暂无话题，下拉刷新或输入你想聊的话题" | 显示通用话题 + 重试按钮 | - |
| Page 3 聊天 | "Leo 正在准备话题..." | 欢迎消息 | 网络错误 + 重试 | - |
| Page 3 生词面板 | - | "对话中点击不认识的单词，释义和例句会出现在这里" | - | - |
| Page 3 查词弹窗 | "查询中..." | - | "无法获取释义" | 展示完整释义卡片 |
| Page 4 复习 | "加载复习队列..." | "还没有生词，去聊个话题吧" | - | 复习完成统计 |

### 交互规范

1. **所有"返回"按钮行为一致**：左上角，箭头图标，点击 = 保存当前状态后返回上一级
2. **弹窗关闭方式**：点关闭按钮、点弹窗外部区域、按 Esc 键——三种方式都生效
3. **按钮防抖**：所有涉及网络请求的按钮，点击后 2 秒内不可重复点击
4. **文本输入**：发送消息后输入框清空，焦点保持在输入框

### Prompt 安全规范

1. **内容过滤**：在 System Prompt 中加入："Do not generate content about politics, violence, or adult topics. If the user brings these up, gently steer the conversation back to the current topic."
2. **话题生成**：LLM 生成的话题不涉及政治、色情、暴力内容
3. **角色一致性**：AI 不应该突然切换人格或语言风格

## 三、优先级排序：MVP 必须解决的

### 第一优先（没做就没法用）

1. 🔴 角色创建的事务性（全完成或全不存）
2. 🔴 生词实时保存（每次点击立即写 DB，不等退出）
3. 🔴 话题生成的 fallback 机制（LLM 挂了也有备用话题）
4. 🔴 自定义话题入口（用户不想聊推荐话题时的出口）
5. 🔴 查词的三级降级（本地 → LLM → 失败提示）
6. 🔴 LLM 超时处理和重试
7. 🔴 角色删除的级联数据清理

### 第二优先（影响体验但可以先上线）

8. 🟡 角色编辑（修改水平/兴趣）
9. 🟡 同一个话题再聊的提示
10. 🟡 每次复习上限 20 张卡片
11. 🟡 复习结束统计
12. 🟡 意外退出保护提示

### 第三优先（可以后续迭代）

13. 🟢 角色数量上限
14. 🟢 话题已读/已聊标记
15. 🟢 聊天记录保留
16. 🟢 水平自动调整
17. 🟢 数据导出

## 四、一个重大设计决策需要你确认

**复习按钮：两个还是四个？**

传统 SM-2 用 4 个按钮（0-5 质量评分）：
```
[完全忘了] [记得很吃力] [想起来了] [太简单了]
```

你说的是两个按钮。我建议折中——用三个按钮：

```
[没想起来]  [有点难]  [想起来了]
    0           2          4
```

原因：两个按钮太粗糙。"有点难"是一个重要的中间状态——用户确实想起来了但很勉强，这种情况应该缩短复习间隔但不重置到零。如果你坚持两个按钮也完全可以，就是 SM-2 的简化版。

你觉得三个按钮可以吗？
